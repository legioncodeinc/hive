# Security Audit: feature/prd-011-onboarding-tenancy (uncommitted working tree)

> Category: Security Audit | Version: 1.0 | Date: July 2026 | Status: Active

Audit of the uncommitted working-tree changes in the hive repo implementing PRD-011: the tenancy onboarding step (org/workspace pickers plus the create-workspace affordance), the active-tenancy display in the shell chrome and nectar panel, and the portal-gate tenancy check (health then auth then tenancy precedence). Performed by security-worker-bee, before quality-worker-bee (pre-flight confirmed: `library/requirements/backlog/prd-011-onboarding-tenancy-selection/qa/` is empty and `library/qa/` holds no QA report for this branch newer than HEAD `b1ba031`).

**Scope note:** the target is the hive portal (React dashboard plus Hono loopback daemon acting as a BFF), not the Hivemind CLI/MCP stack the security-stinger catalogs verbatim. The stinger's threat classes were adapted to this surface: XSS on remote-controlled strings, open redirect, redirect loops, SSRF via the proxy leg, CSRF, secret/token exposure in bodies and logs, telemetry PII posture, hidden-Unicode, and dependency advisories. There is no SQL or captured-trace surface in this branch.

## Executive summary

**No Critical or High findings. No remediation edits were required; the working tree is unchanged by this audit.** One Medium posture finding (CSRF on the two new state-changing `/setup/tenancy` POSTs riding the BFF proxy) is documented, not fixed: it exactly matches the pre-existing posture of every state-changing endpoint on this surface, was documented in the 2026-07-04 PRD-010 audit as F-4, and the correct fix remains the systemic BFF-proxy Origin check that exceeds this branch's blast radius. Two Low findings and one Info observation are documented. Gates: `npm run typecheck` clean, `npm test` 423 passed (423/423), `npm audit --omit=dev` 0 vulnerabilities, hidden-Unicode scan of all changed files clean, no secret-shaped strings in the new tests.

## Audit-focus categories checked

1. **XSS on remote-controlled strings: None detected.** Org and workspace names (attacker-controlled by any workspace member), tenancy labels, and daemon error strings render exclusively as React text children or JSX attribute values, which React escapes: the picker rows and confirm screens (`tenancy-step.tsx:209,221,232,255` via `entityLabel`, `<code>`/`<p>`/`Button` text children; `Button` renders `{children}` as plain children, `primitives.tsx:120`), the shell chrome readout (`active-tenancy-display.tsx:89`, `{text}` span child), the nectar panel line (`pages/hive-graph.tsx`, `Writing to: {panelTenancyLine}` text child), and both error surfaces (`tenancy-step.tsx:177,304`, `{view.message}` / `{selectError}` text children, including daemon-derived `redactedReason` strings). `data-org-id` / `data-workspace-id` ride as React attributes (escaped). Zero occurrences of `dangerouslySetInnerHTML`, `innerHTML`, `document.write`, or `eval` under `src/dashboard/web/` (repo-wide grep).
2. **The gate: None detected.** (a) Open redirect: all three redirect targets are hard-coded `as const` literals (`/buzzing`, `/login`, `/onboarding`, `gate.ts:44-54`) passed directly to `c.redirect(...)`; no request-derived value (path, query, Referer, Host) reaches any Location header, and the g-AC-11 test extends to the tenancy leg including a traversal-shaped path. (b) Exemption set: `GATE_EXEMPT_ROUTES` is an exact-match `includes(pathname)` check (`gate.ts:101-103`) over the three screen literals; `/onboarding` was already exempt pre-branch (PRD-009), and no dashboard route is exempted (prefix exemptions cover only `/api/`, `/setup/`, `/fonts/`, `/assets/` data-plane paths, unchanged by this branch). (c) Fail-closed: `fetchTenancySelected` returns `false` on non-loopback base, network error, abort, non-OK, malformed JSON, or schema mismatch (`setup-tenancy.ts:30-53`), so a tenancy-fetch failure redirects the dashboard to `/onboarding` and never serves it; tests tg-AC-1..4 pin all four outcomes. (d) Loop: `/onboarding` is gate-exempt, so the server can never redirect it (the structural g-AC-9 guarantee extends to the third target); the client-side fault-mode loop is F-2 above. (e) SSRF: the tenancy fetch target is `resolveDaemonBases().honeycomb` (registry entries zod-URL-validated and loopback-filtered at parse, `registry.ts:34-50`) re-checked with `isLoopbackBaseUrl` before the fetch, path is the fixed literal `/setup/tenancy`, and `redirect: "error"` pins the fetch, mirroring `setup-auth.ts` exactly.
3. **The BFF proxy leg: None detected.** `proxy.ts` is untouched by this branch. The five tenancy paths are fixed literals in `ENDPOINTS` (`wire.ts:466-469`) and `tenancy-client.ts`; the only request-derived value is the org id, which rides as `?org=` through `encodeURIComponent` (query string, never the path, never an origin). Owner resolution keys on the URL-parser-normalized pathname (only the `/api/hive-graph` prefix selects nectar; `/setup/*` always resolves honeycomb), so no crafted path can widen the proxy to a different origin, and the resolved base is loopback-checked twice with `redirect: "error"` pinned. Request bodies are exactly `{ orgId, workspaceId }` and `{ org, name }`; response bodies carry ids and names only (the honeycomb PRD-073c contract keeps the short-lived token in daemon memory and never on any response body, and errors flow through `redactedReason`, which slices the message and never the token). No `console.*` in any onboarding module; every failure path swallows silently, so nothing is logged client-side.
4. **The create-workspace input: mirrors the contract; None beyond F-3.** The name is trimmed, required non-empty (matching the contract's `min(1)`), enters the request only via `JSON.stringify` (no concatenation into path or query), and the submit is disabled while busy or empty. The honeycomb side slugifies to the `^[a-z0-9]+(?:[-_][a-z0-9]+)*$` <= 34 id pattern and validates the body with zod. The missing upper bound is F-3 (contract-side, documented).
5. **Funnel telemetry: None detected; posture exceeded.** The new events carry NO org or workspace names and NO ids: `tenancy_selected` sends only a bucketed org count (`single|few|many`) and a boolean `singleOrgConfirm`; `tenancy_shown` and `workspace_created` carry no properties (`tenancy-step.tsx:124-127,146`). This is stricter than the existing posture (which permits ids). All three ride the same fire-and-forget `sendEvent` chokepoint as the PRD-009c events. Daemon-side, the closed schema currently rejects them (F-4, fail-closed), and even for accepted events the emitter forwards only fixed extras (`funnel-telemetry.ts:143-162`), so arbitrary UI-supplied properties can never reach telemetry.

## Additional categories checked (stinger catalog, adapted)

- **Credential/token exposure:** no token, JWT, or credential appears in any new request body, response schema, log statement, or test fixture (grep of the four new test files for token/secret/password/bearer/jwt: clean). The tenancy calls deliberately carry no onboarding token (ts-AC-12), and `wire.ts` tenancy methods add only the fixed non-secret `DASHBOARD_SESSION_HEADERS` identifiers. None detected.
- **Zod wire boundaries:** every tenancy schema is fail-soft (`.catch()` defaults at every field); a malformed or hostile body degrades to `UNSELECTED_SETUP_TENANCY` / empty lists / `null`, never a throw into React (`tenancy-contracts.ts`, `tenancy-client.ts`, `wire.ts:526-555`). The gate-side schema is `selected: z.boolean().catch(false)`, fail-closed. None detected.
- **Hidden Unicode / rules-file backdoor:** PCRE scan of all seven changed/new source files for zero-width, bidi, and word-joiner code points: clean.
- **Dependencies:** `npm audit --omit=dev`: 0 vulnerabilities. No new dependencies added by this branch.
- **Secrets committed:** no new secret-shaped literals in the diff or the untracked files. None detected.
